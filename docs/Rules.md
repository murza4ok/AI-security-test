# Development Rules

Правила разработки и сопровождения проекта `ai-sec`.

---

## 1. Язык и стиль

1. Комментарии в коде, обучающий контент и пользовательская справка пишутся на русском языке.
2. Имена типов, функций, модулей, полей и CLI-ключей остаются на английском языке.
3. Новые документы проекта должны сохраняться в UTF-8 без битой кодировки.

---

## 2. Архитектурные правила

4. `main.rs` — только точка входа. Прикладная orchestration-логика живёт в `src/app/*`.
5. UI не должен влиять на core engine. `cli/*` и interactive flow не должны диктовать форму runtime-моделей.
6. Реализации атак не работают с HTTP напрямую. Все запросы идут только через `LLMProvider`.
7. Классические атаки по возможности должны использовать общий execution path, а не копировать раннер по файлам.
8. Scenario mode и classic mode могут различаться по источнику контекста, но должны сходиться в единый `AttackResult`.

---

## 3. Модель данных

9. `AttackResult` должен оставаться компактным и стандартизированным:
   - базовые поля payload/response;
   - `evaluation`;
   - `evidence`;
   - `damage`.
10. Новые attack mode-ы не должны добавлять плоские ad-hoc поля в `AttackResult`, если это можно выразить через `evidence` или отдельную report transform-логику.
11. Ущерб оценивается по единой модели `H1-H3`.
12. JSON-отчёт должен оставаться versioned и backward-compatible настолько, насколько это разумно для чтения старых результатов.

---

## 4. Generator mode

13. Генератор работает от curated seed-корпуса, а не от полностью случайных строк.
14. Генератор не должен валить весь attack run при локальной ошибке, плохом JSON или таймауте отдельного seed.
15. Генерация должна быть ограничена бюджетом времени и фиксироваться в session metadata.
16. Если генераторное поведение влияет на воспроизводимость, это должно отражаться в отчёте.

---

## 5. Scenario mode

17. Scenario fixtures должны оставаться synthetic и безопасными для хранения в репозитории.
18. Изменения в fixtures нужно рассматривать как изменения данных эксперимента, а не как мелкую косметику.
19. Scenario evaluator должен использовать ту же унифицированную damage model, что и остальные режимы.

---

## 6. Провайдеры

20. API-ключи и токены берутся только из `.env` или env vars. Хардкод запрещён.
21. `health_check()` должен проверять реально полезную работоспособность провайдера, а не только доступность endpoint-а.
22. Общая retry/backoff policy должна жить в shared provider layer.

---

## 7. Качество кода

23. Перед удалением старого блока нужно убедиться, что он не используется через grep/поиск по коду.
24. `dead_code`, transitional wrappers и compatibility-хвосты допускаются только если они реально нужны для миграции.
25. Не использовать `unwrap()` в production-path без жёсткого инварианта и понятного `expect(...)`.
26. Каждый модуль должен иметь внятный `//!`-комментарий, если он является архитектурно значимым.

---

## 8. Тестирование

27. Любой значимый рефакторинг завершается как минимум `cargo test`.
28. Изменения в provider logic и generator mode должны сопровождаться тестом или явной ручной проверкой.
29. Изменения в сценарном режиме должны проверяться не только unit-тестами, но и коротким end-to-end прогоном, если провайдер доступен.

---

## 9. Git и публикация

30. Не коммитить реальные секреты, реальные документы и реальные ответы продуктивных моделей.
31. Локальные пользовательские файлы и личные заметки не включаются в commit без явного запроса.
32. Перед пушем в оба remote нужно убедиться, что в staging нет случайных fixture-изменений, если они не были частью задачи.

---

## 10. Документация

33. `docs/` и корневые технические `.md` файлы считаются живой документацией, а не архивом.
34. `docs/Memory.md` и `docs/Plan_for_operator.md` считаются устаревшим подходом и не должны возвращаться в репозиторий.
35. Каждая feature-ветка обязана проверить и при необходимости обновить документы в своём scope, даже если изменения не затрагивают CLI напрямую.
36. Если меняются команды запуска, флаги, API, schema, сценарии или формат отчёта, документация обновляется в той же ветке без откладывания “на потом”.
37. Перед финальной интеграцией должен выполняться отдельный docs consistency sweep по `README.md`, `docs/*`, `Architecture.md`, `TZ.md` и рабочим планам.
38. Все пользовательские команды в документации должны использовать реальный способ запуска бинарей: сейчас это `cargo run --bin ai-sec -- ...` и `cargo run --bin web_target -- ...`, если не указано иное.
