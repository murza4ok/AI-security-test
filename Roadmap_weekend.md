# Roadmap Weekend

## 1. Назначение документа

Этот план объединяет:

- целевое состояние проекта из [TZ.md](./TZ.md);
- фактические технические проблемы из [refactoring.md](./refactoring.md);
- уже существующий задел по `ai-sec`, scenario-driven режиму и `web_target`.

План предназначен как практическая дорожная карта на ближайшую итерацию разработки, после которой можно запускать поэтапную модификацию проекта.

## 2. Цель weekend-итерации

К концу итерации проект должен перейти из состояния “рабочий, но неоднородный исследовательский прототип” в состояние “стабилизированная база для второй статьи и дальнейшего feature-development”.

Ожидаемый результат:

- документация соответствует фактическому запуску;
- ядро `ai-sec` воспроизводимо и предсказуемо;
- scenario-driven режим не содержит ложных фич;
- `web_target` оформлен как полноценная целевая подсистема, а не как побочный прототип;
- сформирован фундамент для HTTP-атак и multi-turn logic;
- закрыты самые опасные архитектурные долги, мешающие дальнейшей разработке.

## 3. Основные принципы выполнения

1. Сначала исправляется честность и воспроизводимость текущего ядра, затем добавляются новые возможности.
2. Ложные интерфейсы и полуреализованные фичи либо доводятся до рабочего состояния, либо удаляются.
3. Любая новая feature-ветка должна опираться на детерминированное поведение и корректные отчёты.
4. Web-target и attacking engine должны оставаться архитектурно разделёнными.
5. Документация должна обновляться одновременно с поведением кода.

## 4. Главные блокеры из аудита

Перед активным feature-work необходимо закрыть следующие узкие места:

- README и реальные команды запуска расходятся после появления второго бинаря.
- `--model` сейчас может некорректно применяться ко всем провайдерам сразу.
- Scenario reports не сохраняют реальный envelope, отправленный модели.
- `session_seed`, `supports_system_prompt`, `verbose`, `AttackConfig.system_prompt` выглядят как фичи, но фактически не работают.
- Порядок payload-ов и часть scenario logic недетерминированы или неполно реализованы.
- В коде заметное дублирование в provider factory, OpenAI-compatible providers, classic attack family и runtime orchestration.

## 5. Обязательная линия работ

Ниже перечислен обязательный объём работ, который имеет смысл реализовывать в первую очередь.

### Этап 0. Документация и DX-контракт

Цель:

- зафиксировать единый корректный способ запуска и использования проекта.

Задачи:

- определить и внедрить единый способ запуска CLI:
  - либо `default-run = "ai-sec"` в `Cargo.toml`,
  - либо полный переход документации на `cargo run --bin ai-sec -- ...`;
- обновить `README.md`, `Architecture.md`, `docs/*`, примеры команд и smoke-path;
- проверить, что `check`, `list`, `run`, `review`, `compare` документированы честно;
- синхронизировать описания режимов `educational`, `generative`, `scenario-driven`, `web-target`.

Критерий завершения:

- любой пример запуска из основной документации работает без ручного “угадывания”, какой бинарь нужно выбрать.

### Этап 1. Стабилизация ядра `ai-sec`

Цель:

- убрать поведенческие сюрпризы из текущего runtime.

Задачи:

- исправить семантику `--model`, чтобы override применялся предсказуемо и безопасно;
- сделать deterministic payload loading:
  - сортировка файлов;
  - стабильный порядок payload-ов;
- определить и реализовать честную семантику `--limit`;
- убрать или реализовать `verbose`;
- убрать или реализовать `AttackConfig.system_prompt`;
- пересмотреть panic-paths и заменить их корректными ошибками там, где это нужно.

Критерий завершения:

- CLI behaves predictably, а базовые прогоны больше не зависят от случайного порядка файлов или неявных override-ов.

### Этап 2. Приведение scenario subsystem к реальному контракту

Цель:

- сделать scenario-driven режим воспроизводимым и честным относительно манифеста и отчётности.

Задачи:

- определить, какие поля `scenario.toml` реально участвуют в логике;
- привести schema и implementation к одному источнику истины;
- либо реально реализовать `session_seed`, `mask_pii`, `credential_patterns`, threat-model-driven behavior,
  либо убрать их из пользовательского контракта текущей версии;
- сделать deterministic retrieval subset;
- исключить двойную загрузку сценария в runtime;
- сохранить в JSON-отчёте реальный request envelope:
  - system prompt;
  - hidden context mode;
  - retrieved documents metadata;
  - user-context mode;
- унифицировать сборку scenario metadata на session-level.

Критерий завершения:

- `sensitive_data_exposure` становится воспроизводимым режимом, а не смесью данных, эвристик и частично задокументированных обещаний.

### Этап 3. Рефакторинг архитектурного дублирования

Цель:

- сократить объём копипасты, который уже мешает развитию проекта.

Задачи:

- объединить provider factory в один внятный путь сборки;
- вынести общую логику OpenAI-compatible providers;
- убрать дублирование циклов “run per provider + save report”;
- сократить шаблонный код classic attack family;
- уменьшить количество `#[allow(dead_code)]` и мёртвых helper-функций;
- пересобрать derived metrics так, чтобы summary имел один источник истины.

Критерий завершения:

- добавление нового провайдера, attack family или режима не требует дублировать существующий код блоками по 50–100 строк.

### Этап 4. Доведение `web_target` до статуса полноценной цели

Цель:

- превратить существующий web-target из демо-прототипа в управляемую целевую подсистему.

Задачи:

- зафиксировать архитектуру web-target:
  - auth/session;
  - chat orchestration;
  - policy layer;
  - tool layer;
  - state/data access;
  - redaction/audit;
- выделить policy decisions из длинной substring-цепочки в более структурированный слой;
- формализовать различие `naive`, `segmented`, `guarded`;
- унифицировать работу с customer scope, tenant scope, internal notes и raw records;
- подготовить данные и ответы web-target к последующему автоматическому анализу;
- сделать API-контракт устойчивым для `ai-sec` HTTP-client.

Критерий завершения:

- web-target можно рассматривать как реальную демонстрационную цель, а не просто как локальный чат с жёстко вшитыми if/else.

### Этап 5. HTTP-интеграция `ai-sec` -> `web_target`

Цель:

- обеспечить атаку web-контура через HTTP из основного инструмента.

Задачи:

- спроектировать target client / provider для `/api/chat`;
- реализовать session persistence и login flow;
- добавить CLI-флаги для:
  - base URL,
  - пользователя,
  - security profile;
- сохранять в отчёте:
  - profile,
  - authenticated user,
  - tenant,
  - tool calls attempted/allowed/denied,
  - redactions,
  - leak type;
- обеспечить совместимость существующих attack family с HTTP target mode.

Критерий завершения:

- `ai-sec` умеет атаковать web-target без ручного участия человека в браузере.

### Этап 6. Multi-turn и adaptive attack foundation

Цель:

- подготовить проект к главной идее из ТЗ: атаки, развивающиеся в зависимости от ответа цели.

Задачи:

- ввести conversation chain abstraction;
- поддержать stateful request sequences;
- описать payload chain format;
- реализовать минимум базовый сценарий:
  - warm-up,
  - escalation,
  - extraction,
  - confirmation;
- определить chain-level evaluator contract;
- подготовить связку между generator mode и промежуточным ответом цели.

Критерий завершения:

- проект получает реальный фундамент для agentic и adaptive атак, а не только single-shot payload mutation.

### Этап 7. Reporting, тесты и демонстрационная готовность

Цель:

- сделать проект пригодным для статьи, demo и дальнейшего безопасного расширения.

Задачи:

- расширить terminal report и JSON report под HTTP-target и multi-turn сценарии;
- покрыть тестами:
  - provider factory;
  - deterministic payload loading;
  - scenario loader/builder/retrieval/evaluator;
  - web-target policy;
  - HTTP session flow;
- проверить сценарии на `Ollama`;
- сформировать минимальный demo-matrix:
  - `support_bot`;
  - `hr_bot`;
  - `internal_rag_bot`;
  - минимум 2 модели;
- убедиться, что сравнение профилей и сценариев читаемо в отчётах.

Критерий завершения:

- проект можно запускать как reproducible demo lab и использовать как техническую основу второй статьи.

## 7. Реалистичный объём на ближайшую итерацию

Если придерживаться практичного weekend-scope, то обязательный минимум должен быть таким:

1. Починить способ запуска и документацию.
2. Исправить `--model`, `--limit`, payload ordering и другие P1 runtime-проблемы.
3. Привести scenario subsystem к честному контракту.
4. Убрать наиболее болезненное дублирование в provider factory и runtime.
5. Привести `web_target` к стабильному API и внутренней структуре, на которую можно опереться в следующем шаге.

Это создаст хорошую базу для следующей итерации, в которой уже можно будет спокойно внедрять:

- полноценный HTTP attack mode;
- multi-turn payload chains;
- response-aware generative attacks.

## 8. Stretch scope

Если после обязательной линии останется ресурс, можно расширить объём работ:

- объединить OpenAI и DeepSeek в один OpenAI-compatible provider layer;
- перевести classic attack family в более декларативную форму;
- реализовать начальную версию chain-level evaluator;
- добавить benchmark profiles `quick`, `baseline`, `full`;
- усилить `guarded` profile через response validation и audit trail.

## 9. Что не стоит делать раньше времени

- Не надо строить сложный frontend раньше стабилизации backend contract.
- Не надо добавлять новые attack family, пока текущие режимы частично документированы и частично реализованы.
- Не надо углубляться в embeddings/RAG sophistication до наведения порядка в deterministic retrieval.
- Не надо развивать agentic-слой, пока single-session и HTTP-session metadata ещё не стабилизированы.

## 10. Конечный ожидаемый эффект

После прохождения этой дорожной карты проект должен стать:

- архитектурно чище;
- честнее по документации;
- устойчивее для расширения;
- сильнее как демонстрационный AI-security стенд;
- пригодным для запуска дальнейшей модификации силами агента без постоянной борьбы с накопленным техдолгом.
