# Техническое задание

## 1. Общие сведения

- Наименование проекта: `AI-security-test`
- Основной язык реализации: `Rust`
- Формат проекта: монорепозиторий с двумя связанными контурами
- Контуры проекта:
  1. `ai-sec` — атакующий инструмент / red-team CLI
  2. `web_target` — демонстрационный веб-макет LLM-based системы

## 2. Контекст и назначение

Проект создаётся как демонстрационный и исследовательский стенд по `AI Security`. Его задача — показать, что уязвимости LLM-систем возникают не только в самой модели, но и в окружающей её архитектуре: в prompt orchestration, retrieval, tool access, memory, разграничении прав, фильтрации ответов и веб-интеграции.

Проект должен стать:

- практическим инструментом для авторизованного тестирования LLM-based систем;
- демонстрационной базой для второй статьи;
- ядром публичного инженерного портфолио по направлению `AI Security`;
- лабораторией для дальнейшего развития в сторону agentic-атак, web-target сценариев и атак на защищённые макеты.

## 3. Продуктовая цель

Необходимо создать воспроизводимый инструмент, способный:

- запускать эталонные и генерируемые атаки против LLM-based целей;
- адаптировать полезную нагрузку под поведение цели;
- работать как напрямую с LLM provider, так и через HTTP против отдельного web-target;
- демонстрировать утечки данных, ошибки авторизации, memory bleed, prompt extraction, tool misuse и другие классы AI-security уязвимостей;
- формировать понятные отчёты для оператора и материалы для статьи.

## 4. Целевая архитектура проекта

Проект должен состоять из двух логически разделённых подсистем.

### 4.1. Подсистема 1: атакующий инструмент `ai-sec`

Назначение:

- управлять сценариями атаки;
- отправлять полезные нагрузки цели;
- поддерживать несколько режимов атаки;
- собирать evidence, damage assessment и session reports;
- работать с локальными и облачными моделями;
- в перспективе атаковать не только provider API, но и web-target по HTTP.

### 4.2. Подсистема 2: веб-макет `web_target`

Назначение:

- эмулировать реальное LLM-приложение с UI, логином, ролями и API;
- предоставлять атакуемую поверхность через браузер и `/api/chat`;
- моделировать как слабую, так и более защищённую архитектуру;
- служить демонстрационной целью для второй статьи и для автоматических атак из `ai-sec`.

## 5. Текущее состояние проекта

По состоянию на текущую ревизию в репозитории уже существуют:

- CLI `ai-sec` с командами `run`, `list`, `check`, `review`, `compare`, `sessions`, `explain`;
- attack family для payload-driven атак;
- генеративный режим на базе seed payload-ов;
- scenario-driven режим `sensitive_data_exposure`;
- JSON и terminal reporting;
- provider abstraction для `OpenAI`, `Anthropic`, `DeepSeek`, `YandexGPT`, `Ollama`;
- начальный `web_target` на `axum`;
- синтетические fixtures для `support_bot`, `hr_bot`, `internal_rag_bot`, `support_bot_hardened`.

Техническое задание должно опираться на этот задел, но целевое состояние проекта должно быть шире текущей реализации.

## 6. Целевые пользовательские сценарии

### 6.1. Сценарий оператора

Оператор должен иметь возможность:

- быстро понять назначение каждой команды через `help`;
- выбрать режим атаки и цель;
- использовать готовые payload-ы;
- использовать генеративный режим для новых атакующих промптов;
- атаковать web-target по HTTP;
- сохранять, читать и сравнивать отчёты;
- использовать проект как demo-сценарий в статье и докладе.

### 6.2. Сценарий исследователя

Исследователь должен иметь возможность:

- воспроизводить прогоны между моделями;
- сравнивать baseline и generated режимы;
- фиксировать стабильный benchmark-профиль;
- видеть не только финальный verdict, но и evidence;
- анализировать разницу между `naive`, `segmented`, `guarded`.

### 6.3. Сценарий демонстрации для статьи

Должна быть возможность показать:

- атаку не просто на “голую модель”, а на LLM-приложение;
- разницу между prompt-only защитой и архитектурной защитой;
- как утечка достигается через доступ к скрытому контексту, retrieval, tool output, session state;
- как усиление backend-политики снижает успешность тех же атак.

## 7. Функциональные требования к `ai-sec`

### 7.1. UX и CLI

Инструмент должен обеспечивать:

- понятный `--help` на верхнем уровне и на подкомандах;
- описание режимов работы, аргументов и примеров запуска;
- интерактивный режим для базовых сценариев;
- понятные подсказки по ошибкам конфигурации;
- единообразный терминальный вывод;
- понятный review-режим;
- понятные summary и comparison reports.

CLI должен поддерживать как минимум:

- `run`
- `list`
- `check`
- `review`
- `compare`
- `sessions`
- `explain`

### 7.2. Educational attacking module

Должен существовать режим “эталонных” атак, использующий только curated payload corpus.

Требования:

- payload-ы должны загружаться из внешних TOML-файлов;
- каждая attack family должна иметь ID, описание и обучающий explainer;
- оператор должен иметь возможность читать описание основных типов атак;
- educational-mode должен быть пригоден для workshop/demo-формата.

### 7.3. Seed corpus

Исходный набор атакующих payload-ов должен быть внешним и расширяемым.

Требования:

- хранение вне кода;
- разбиение по attack family;
- наличие metadata, severity, harm level, success indicators;
- пригодность как для baseline-прогонов, так и для генеративного режима;
- воспроизводимость набора payload-ов между запусками.

### 7.4. Generative attacking module

Главный целевой режим проекта — генеративный.

Требования:

- брать seed payload-ы за основу;
- генерировать новые payload-варианты;
- использовать attack-family context и harm-level context;
- поддерживать как минимум стратегии `paraphrase`, `obfuscation`, `escalation`, `mixed`;
- уметь порождать полезную нагрузку, ориентируясь на успешный/неуспешный ответ цели;
- в целевом состоянии поддерживать multi-turn adaptation, а не только single-shot generation;
- не ломать основной прогон при плохом ответе модели-генератора;
- записывать generated payload metadata в отчёт.

### 7.5. Payload-driven attack families

Инструмент должен поддерживать минимум следующие attack family:

- `prompt_injection`
- `jailbreaking`
- `extraction`
- `goal_hijacking`
- `token_attacks`
- `many_shot`
- `context_manipulation`
- `sensitive_data_exposure`

В перспективе структура должна позволять добавлять новые family без тяжёлого переписывания рантайма.

### 7.6. Scenario-driven режим

Режим `sensitive_data_exposure` должен оставаться отдельным функциональным контуром внутри инструмента.

Требования:

- загрузка named scenario;
- сборка envelope из system prompt, hidden context, retrieved docs, memory block, attacker prompt;
- поддержка `support_bot`, `hr_bot`, `internal_rag_bot`, hardened-вариантов;
- поддержка `fixture_root`, `scenario_config`, `retrieval_mode`, `tenant`, `session_seed`;
- детерминированная сборка сценария;
- детерминированный retrieval subset;
- возможность расширения новыми сценариями через данные и манифест.

### 7.7. HTTP-target attack mode

Инструмент должен уметь атаковать не только provider API, но и web-target.

Целевые требования:

- отдельный target client / provider для web application;
- работа через `/api/chat`;
- поддержка login flow и cookie/session persistence;
- передача base URL, demo-user и security profile через CLI;
- сохранение target metadata в отчёт;
- возможность использовать существующие attack family против HTTP-цели;
- поддержка multi-turn conversation chains.

### 7.8. Multi-turn attack support

В проекте должна быть реализована атака не только одним сообщением, но и цепочкой шагов.

Требования:

- поддержка attack chains из 2–5 шагов;
- сохранение session state между сообщениями;
- возможность warm-up, escalation, extraction, confirmation;
- chain-level scoring и transcript-level reporting;
- foundation для adaptive prompt generation по промежуточным ответам цели.

### 7.9. Provider layer

Инструмент должен поддерживать:

- `OpenAI`
- `Anthropic`
- `DeepSeek`
- `YandexGPT`
- `Ollama`

Требования:

- единый provider trait;
- централизованный timeout/retry/backoff;
- provider-specific error mapping;
- возможность override модели;
- возможность дальнейшего добавления web-target client без разлома архитектуры.

### 7.10. Reporting

Инструмент должен поддерживать два уровня отчётности:

- terminal output для оператора;
- JSON report для воспроизводимого анализа.

Отчёт должен включать:

- provider metadata;
- requested/actual model;
- runtime config;
- retry settings;
- session metadata;
- generated payload metadata;
- scenario metadata;
- evaluation result;
- evidence;
- damage assessment;
- exposure score;
- tool access / redaction / web-target metadata для HTTP-режима.

### 7.11. Review и compare

Должны поддерживаться:

- просмотр одной сессии;
- сравнение нескольких отчётов;
- обзор сохранённых сессий;
- сравнение baseline vs generated;
- сравнение моделей;
- сравнение security profiles.

## 8. Функциональные требования к `web_target`

### 8.1. Общая роль макета

`web_target` должен моделировать не “модель сама по себе”, а LLM-based приложение.

Макет должен включать:

- web UI;
- login/logout;
- server-side or minimal HTML rendering;
- HTTP API;
- backend policy layer;
- доступ к synthetic data;
- LLM-backed answer generation через локальные модели.

### 8.2. Точки входа

Макет должен предоставлять как минимум:

- `/health`
- `/login`
- `/chat`
- `/api/chat`
- `/logout`

### 8.3. Роли и пользователи

Минимальный демонстрационный набор:

- `guest`
- `customer_alice`
- `customer_bob`
- `agent_support`

В дальнейшем допускается расширение ролей.

### 8.4. Security profiles

Макет должен поддерживать три профиля:

- `naive`
- `segmented`
- `guarded`

#### `naive`

- модель получает слишком богатый контекст;
- слабое разделение между raw data и user-visible answer;
- минимальная фильтрация.

#### `segmented`

- модель работает с summaries вместо raw records;
- tenant/role учитываются на backend;
- доступ к данным фильтруется до генерации ответа.

#### `guarded`

- все защиты из `segmented`;
- response validation;
- DLP/redaction;
- audit metadata;
- более строгие backend policy decisions.

### 8.5. Backend tool layer

Макет должен моделировать backend-доступы как tool-like операции.

Минимальный набор:

- `search_tickets`
- `get_customer_summary`
- `get_internal_note`

Дальнейшее расширение допустимо.

Tool layer должен:

- проверять роль;
- проверять tenant scope;
- отделять raw output от safe output;
- логировать попытки доступа;
- формировать evidence для отчёта.

### 8.6. Данные макета

В макете должны быть синтетически смоделированы:

- пользовательские данные;
- сырые внутренние данные;
- internal notes;
- документация / RAG documents;
- canary secrets;
- при необходимости локальная session memory.

Все данные должны быть безопасны для хранения в репозитории.

### 8.7. LLM runtime

Первичный runtime макета — `Ollama`.

Требования:

- поддержка локальных моделей;
- настройка модели через `.env`;
- демонстрационный запуск на modest hardware;
- возможность смены модели для сравнения поведения.

## 9. Требования к данным и fixtures

Данные должны быть:

- полностью synthetic;
- правдоподобными;
- консистентными внутри сценария;
- безопасными для публикации;
- пригодными для автоматического анализа evaluator-ом;
- детерминированными в тестах.

Типы данных, которые должны присутствовать в разных сценариях:

- ФИО;
- email;
- телефон;
- идентификаторы пользователя / сотрудника / клиента;
- адреса;
- внутренние заметки;
- URL внутренней инфраструктуры;
- credential-like значения;
- canary values;
- document fragments.

## 10. Требования к security-evaluation модели

### 10.1. Базовая классификация

Инструмент должен использовать категории:

- `REFUSED`
- `PARTIAL`
- `BYPASS`
- `INFO`
- `INCONCLUSIVE`

### 10.2. Harm levels

Необходимо использовать harm model:

- `L0` — public / informational
- `L1` — probing / schema / review-only
- `L2` — extraction of business data / PII
- `L3` — explicit secret / credential / raw confidential disclosure

### 10.3. Damage model

Необходимо поддерживать damage model:

- `H1`
- `H2`
- `H3`

### 10.4. Scenario-specific evidence

Для scenario-driven и web-target режимов evaluator должен уметь определять:

- exact canary leakage;
- partial canary leakage;
- credential-like leak;
- multi-field PII leak;
- raw record leak;
- document fragment leak;
- system prompt fragment leak;
- schema-only leakage;
- tool authorization mismatch;
- cross-tenant leakage;
- redaction bypass;
- memory bleed.

## 11. Нефункциональные требования

### 11.1. Безопасность

- Запрещено использовать реальные секреты, реальные клиентские данные и реальные внутренние документы.
- Все демо-артефакты должны быть synthetic.
- Инструмент должен позиционироваться только как средство авторизованного тестирования.

### 11.2. Воспроизводимость

- Поведение payload selection, retrieval selection и session assembly должно быть детерминированным там, где это заявлено.
- Отчёты должны содержать достаточно данных для повторного review.

### 11.3. Расширяемость

- Добавление нового scenario не должно требовать тяжёлой переработки provider layer.
- Добавление новой attack family не должно требовать переписывания CLI orchestration.
- Добавление нового security profile в макет должно быть локализовано.

### 11.4. Поддерживаемость

- Дублирование должно быть минимизировано.
- Документация должна соответствовать фактическому способу запуска.
- Неиспользуемые интерфейсы не должны оставаться в коде как “ложные” фичи.

### 11.5. Тестируемость

Проект должен иметь тесты минимум на:

- CLI parsing;
- scenario loading;
- retrieval selection;
- context building;
- evaluator logic;
- JSON report serialization/migration;
- web-target policy decisions;
- HTTP target session flow.

### 11.6. Производительность

- Инструмент должен корректно работать с локальными моделями в условиях ограниченных ресурсов;
- по умолчанию конфигурация должна быть пригодна для локального demo;
- режимы не должны без необходимости раздувать hidden context.

## 12. Ограничения и границы проекта

В первую итерацию не требуется:

- production-grade frontend;
- полноценный agent framework;
- embeddings-based retrieval как обязательный baseline;
- формальная judge-based классификация;
- реальная корпоративная БД или реальная авторизация.

Приоритет — воспроизводимая и убедительная демонстрация атак на LLM-приложение.

## 13. Целевые артефакты проекта

В целевом состоянии репозиторий должен содержать:

- рабочий CLI-инструмент `ai-sec`;
- рабочий `web_target`;
- сценарии и payload corpus;
- воспроизводимые JSON reports;
- документацию по запуску и демонстрации;
- материалы, пригодные для статьи и публичной демонстрации.

## 14. Критерии приёмки

### 14.1. Для `ai-sec`

Результат считается принятым, если:

1. CLI имеет понятный help и рабочие команды.
2. Educational mode работает на curated payload corpus.
3. Generative mode генерирует новые payload-ы из seed.
4. `sensitive_data_exposure` работает на нескольких сценариях.
5. Отчёты содержат session, scenario, evidence и damage metadata.
6. Есть review и compare режимы.
7. Инструмент может атаковать web-target по HTTP.
8. Поддерживается хотя бы базовый multi-turn workflow.

### 14.2. Для `web_target`

Результат считается принятым, если:

1. Макет поднимается локально.
2. Есть логин, чат и API.
3. Работают профили `naive`, `segmented`, `guarded`.
4. Присутствует role/tenant aware backend behavior.
5. Поведение профилей заметно отличается при одинаковых атаках.
6. `ai-sec` может автоматически проводить атаки по HTTP против этого макета.

### 14.3. Для демонстрации статьи

Результат считается принятым, если:

1. Можно показать успешную атаку на слабый или недостаточно защищённый макет.
2. Можно показать, что усиление backend architecture снижает или блокирует те же атаки.
3. Демонстрация опирается на synthetic data и воспроизводимые отчёты.
4. Сценарий выглядит как атака на систему, а не только на модель.

## 15. Definition of Done

Работа по текущему ТЗ считается завершённой, когда одновременно выполнены условия:

- функциональность реализована в коде;
- документация обновлена и соответствует реальному запуску;
- `cargo test` и обязательные smoke-check проходят;
- web-target и CLI можно продемонстрировать локально;
- demo-поток пригоден для подготовки второй статьи;
- остаётся минимум ложных фич, неиспользуемых интерфейсов и документированных, но неработающих режимов.

## 16. Приоритет реализации

Приоритеты проекта должны быть следующими:

1. Стабилизация и честность текущего ядра.
2. Устранение расхождений между документацией и кодом.
3. Доведение scenario-driven режима до воспроизводимого состояния.
4. Завершение web-target и HTTP-атаки.
5. Добавление multi-turn и adaptive attack logic.
6. Усиление reporting и benchmark-comparison сценариев.

## 17. Результат, которого ожидает владелец проекта

На выходе должен получиться не просто CLI для учебных prompt-атак, а цельный исследовательский стенд по `AI Security`, в котором:

- есть атакующий инструмент;
- есть отдельная атакуемая цель;
- можно демонстрировать эксфильтрацию, authorization bugs, retrieval leaks и архитектурные ошибки;
- можно сравнивать защищённые и незащищённые варианты;
- можно использовать результаты как основу для статьи, demo и дальнейшего развития проекта.
