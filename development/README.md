# Development Orchestration

## Назначение

Этот каталог является step 0 для weekend-итерации.

Его задача:

- превратить `Branch_tasks.md` в набор самодостаточных task-pack для сабагентов;
- зафиксировать правильную иерархию управления разработкой;
- убрать двусмысленность в том, как именно запускать агентов по веткам.

## Правильная модель работы

Неправильная модель:

- несколько пишущих агентов в одной ветке;
- запуск зависимых веток до фиксации базового контракта;
- task-файлы без общего координатора и интеграционной ветки.

Правильная модель:

- один координатор процесса;
- одна интеграционная ветка `codex/weekend-integration`;
- один пишущий агент на одну feature-ветку;
- один проверяющий агент по необходимости после завершения writer-agent;
- merge только после проверок и handoff.

`task.md` — это входной пакет для агента, а не замена review, merge-control и интеграционной проверки.

## Иерархия ролей

### 1. Координатор

Отвечает за:

- порядок запуска веток;
- создание веток от `codex/weekend-integration`;
- выдачу агенту нужного `task.md`;
- решение конфликтов между ветками;
- допуск ветки к merge.

### 2. Интеграционный владелец

Отвечает за:

- состояние `codex/weekend-integration`;
- последовательное слияние готовых веток;
- финальный интеграционный smoke-check.

### 3. Writer-agent

Отвечает за:

- одну feature-ветку;
- работу только в пределах своего scope;
- выполнение обязательных проверок;
- обновление документации в своём scope;
- финальный handoff по ветке.

### 4. Reviewer-agent

Отвечает за:

- проверку готовой ветки;
- поиск багов, регрессий и нарушений scope;
- верификацию того, что документация и код совпадают.

## Дерево управления

1. `main`
2. `codex/weekend-integration`
3. feature-ветки `codex/*`
4. writer-agent на feature-ветку
5. reviewer-agent после завершения writer-agent

## Источники истины

При запуске любого агента источниками истины считаются:

- `Branch_tasks.md` — границы веток и проверки;
- `Roadmap_weekend.md` — общий чек-лист итерации;
- `TZ.md` — целевое техническое состояние;
- `refactoring.md` — технический аудит и проблемные места;
- `README.md`, `Architecture.md`, `docs/*` — текущий рабочий контракт проекта.

## Как запускать writer-agent

1. Создать нужную ветку от `codex/weekend-integration`.
2. Передать агенту только один `task.md`.
3. Явно сказать, что он пишет только в пределах указанного scope.
4. Требовать обновление документации, если меняется контракт.
5. После завершения получить handoff.
6. Запустить reviewer-agent при необходимости.
7. Только потом решать вопрос о merge.

## Что обязательно требовать от writer-agent

- не трогать forbidden scope;
- не откатывать чужие грязные изменения;
- не исправлять “по пути” соседние модули;
- не добавлять лишние сущности вне цели ветки;
- прогнать обязательные проверки;
- вернуть короткий и точный handoff.

## Обязательный handoff по ветке

После завершения любой ветки агент должен отдать:

- какие файлы изменены;
- какие документы изменены;
- какие команды запуска и тесты прогнаны;
- какой ручной smoke-check выполнен;
- какие риски остались;
- готова ли ветка к merge;
- на какой пункт `Roadmap_weekend.md` работала ветка.

## Как использовать task-pack

Каждая папка в `development/branches/*` соответствует одной feature-ветке.

Внутри лежит один `task.md`, который уже содержит:

- branch name;
- зависимость от предыдущих веток;
- цель;
- allowed scope;
- forbidden scope;
- обязательные изменения;
- обязательные проверки;
- условия handoff;
- условия stop/escalate.

## Короткая поправка к подходу

Если говорить строго, не нужно строить “иерархию сабагентов”, которые постоянно живут внутри проекта.

Правильнее думать так:

- есть координатор;
- есть интеграционная ветка;
- есть одноразовые исполнители под конкретные task-pack;
- каждый агент запускается на ограниченную задачу и после handoff считается завершённым.
