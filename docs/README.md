# Documentation Index

## Назначение

`docs/` содержит только живую техническую документацию проекта.

Если документ устарел и больше не поддерживается, он должен быть:

- либо удалён;
- либо заменён актуальным документом;
- либо явно перенесён в другой поддерживаемый источник истины.

## Активные документы в `docs/`

- `Rules.md` — правила разработки, сопровождения и документации.
- `Scenario_Schema_v2.md` — контракт сценарного schema-слоя.
- `Sensitive_Data_Exposure_Spec.md` — спецификация сценарного режима `sensitive_data_exposure`.
- `Ollama_Demo_Setup.md` — практический запуск локального demo-контура на `Ollama`.
- `Ollama_Demo_Generator.md` — генерация demo-harness и вспомогательных скриптов для `Ollama`.

## Активные документы в корне репозитория

- `README.md` — быстрый старт и пользовательский запуск.
- `Architecture.md` — текущее устройство проекта.
- `TZ.md` — целевое техническое задание.
- `Roadmap_weekend.md` — чек-лист ожидаемого результата итерации.
- `Branch_tasks.md` — ветки разработки, их границы и обязательные проверки.
- `refactoring.md` — технический аудит проблем, долга и приоритетов рефакторинга.

## Что больше не используется

Следующие документы выведены из эксплуатации и не должны возвращаться:

- `Memory.md`
- `Plan_for_operator.md`

Причина:

- они были персональными рабочими памятками;
- они быстро устаревают;
- их содержимое теперь должно жить в актуальных документах: `README.md`, `TZ.md`, `Roadmap_weekend.md`, `Branch_tasks.md`, `refactoring.md` и тематических файлах из `docs/`.

## Правило сопровождения

Каждая feature-ветка обязана:

- проверить документы в своём scope;
- обновить примеры команд и ограничения;
- не оставлять расхождение между кодом и документацией.

Перед финальной интеграцией выполняется отдельный `docs consistency sweep`.
